# Philoctetes (Phil) - iOS

A health and fitness iOS app for calorie tracking, workout logging, and fitness provider integration. Built with SwiftUI, Core Data, and a modular SPM architecture.

## Tech Stack

| Layer | Choice |
|---|---|
| UI Framework | SwiftUI + `@Observable` (iOS 26+) |
| Architecture | MVI/Redux + Clean Architecture |
| Persistence | Core Data (offline-first) |
| Networking | URLSession (async/await) |
| DI | Factory (SPM) |
| Auth | Firebase Auth |
| Navigation | Coordinator pattern |
| Modularization | Local SPM packages |

## Architecture

### MVI/Redux Store

Unidirectional data flow: **View → Action → Reducer → State → View**. One-time events (navigation, alerts) flow as **Effects**.

Every feature screen has a `Store<State, Action, Effect>` subclass. Reducers are pure synchronous functions. Async work lives in `handleSideEffects`.

### Clean Architecture Layers

```
Domain (entities, use case protocols, repository protocols)
  ↑
Data (repository implementations, DTOs, mappers, data sources)
  ↑
Presentation (stores, views, coordinators)
```

### Interface + Implementation Split

Each feature is an SPM package with two targets:
- **Interface** — public protocols and entities (e.g., `CalorieTrackingInterface`)
- **Feature** — internal implementations (e.g., `CalorieTrackingFeature`)

Features depend on other features' *interfaces*, never their implementations. The main app target wires everything together via Factory DI.


### Dependency Graph

```
Phil App (thin shell — imports all Feature implementations)
    ↓
Feature Implementations (AuthFeature, CalorieTrackingFeature, etc.)
    ↓
Feature Interfaces (AuthInterface, CalorieTrackingInterface, etc.)
    ↓
Common Modules (CommonDomain, CommonUI, CommonNetwork, CommonPersistence)
```

## Key Patterns

### Store (MVI/Redux)

```swift
@Observable
final class DailyIntakeStore: Store<State, Action, Effect> {
    override func reduce(_ state: inout State, _ action: Action) { /* pure */ }
    override func handleSideEffects(_ action: Action) async { /* async work */ }
}
```

### Use Cases

Protocol in Interface, implementation in Feature:

```swift
// Interface
public protocol LogFoodEntryUseCaseProtocol: Sendable {
    func execute(entry: FoodEntry) async throws -> FoodEntry
}

// Feature
final class LogFoodEntryUseCase: LogFoodEntryUseCaseProtocol { ... }
```

### Repository (Offline-First)

Writes go to Core Data first, then push to API in background if online.

### Coordinator Navigation

Views emit Effects → Coordinator handles navigation. Views never navigate directly.

### Factory DI

```swift
extension Container {
    var foodEntryRepository: Factory<FoodEntryRepositoryProtocol> {
        self { FoodEntryRepository(...) }
    }
}
```

## Testing

### Pyramid

- **Unit tests** — in each SPM package's test target (stores, use cases, repositories, mappers)
- **Integration tests** — in `PhilTests/` (Core Data, network, DI resolution)
- **UI tests** — in `PhilUITests/` (end-to-end flows) + snapshot tests per package

### Running Tests

```bash
# Run all package tests
for pkg in Packages/*/; do
    cd "$pkg" && swift test && cd ../..
done

# Run from Xcode
# Cmd+U to run all tests, or select specific test targets
```

### Mock Generation

Protocols annotated with `/// @mockable` get mocks auto-generated by Mockolo.

## Build & Run

1. Open `Phil/Phil.xcodeproj` in Xcode 26+
2. Select the `Phil` scheme and an iOS 26+ simulator
3. Build and run (Cmd+R)

Local SPM packages are resolved automatically from `Packages/`.
